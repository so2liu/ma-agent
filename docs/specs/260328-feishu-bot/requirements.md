# 需求文档：飞书机器人集成

## 简介

在 Ma-Agent（小马快跑）Electron 桌面端中集成飞书机器人，通过飞书 SDK 的 WebSocket 长连接模式接收消息，使用户可以通过飞书客户端（手机/桌面）向 Pi Agent 发送指令，实现移动端远程控制 Claude Code。

## 背景

当前 Ma-Agent 只能通过桌面端 UI 与 Pi Agent 交互。用户希望在移动端（通勤、外出）也能控制 Agent 完成编码任务。飞书作为用户日常办公工具，是最自然的移动端入口。

## 技术基础

- **SDK**: `@larksuiteoapi/node-sdk`，提供 `WSClient`（WebSocket 事件接收）和 `Client`（API 调用）
- **WebSocket 长连接模式**: SDK 主动连出连接到飞书服务器，无需公网 IP/域名/内网穿透
- **事件自动解密**: WebSocket 模式下收到的事件已预解密，无需手动验签

## 架构概览

```
飞书客户端 → 飞书开放平台 ←WebSocket→ Electron 主进程(SDK WSClient) → AgentRuntime → Pi Agent → Claude Code
```

## 需求

### 需求 1 - WebSocket 连接管理

**用户故事：** 作为用户，我想要 Electron 应用通过 WebSocket 长连接接收飞书消息，以便无需公网 IP 就能使用飞书机器人。

#### 验收标准

1. 当 Electron 应用启动时，如果飞书机器人功能已启用且配置完整，那么系统应该通过 `lark.WSClient` 建立 WebSocket 长连接
2. 当 WebSocket 连接断开时，SDK 应该自动重连（SDK 内置能力）
3. 当 Electron 应用退出时，系统应该优雅关闭 WebSocket 连接

### 需求 2 - 消息接收与格式转换

**用户故事：** 作为用户，我想要在飞书中发送各种类型的消息给机器人，机器人能正确理解并转发给 Pi Agent。

#### 支持的消息类型及转换规则

| 飞书消息类型 | 场景 | 转换为 Agent 的格式 |
|---|---|---|
| `text` | 用户发送纯文本 | 提取 `content.text`，将 `@_user_N` 占位符替换为 `[用户名称](用户ID)`（从 `mentions[]` 中查找） |
| `post` | 用户发送富文本 | 遍历 `content` 段落数组，逐 tag 转 Markdown：`text` → 原文（支持 bold/italic 等 style），`a` → `[text](href)`，`at` → `[name](user_id)`，`img` → 下载图片后以附件形式传递，`code_block` → `` ```lang\ncode``` ``，`emotion` → emoji 名称 |
| `image` | 用户发送图片/截图 | 通过 `client.im.v1.image.get()` 下载到 attachments 目录，作为附件传递给 Agent |
| `file` | 用户发送文件（Excel、PDF、Word 等） | 通过 `client.im.v1.file.get()` 下载到 attachments 目录，作为附件传递给 Agent（保留原始文件名） |
| `media` | 用户发送视频 | 通过 API 下载到 attachments 目录，作为附件传递给 Agent（即使 Agent 可能无法直接播放，也要先保存） |
| `audio` | 用户发送语音消息 | 暂不支持，回复提示"暂不支持语音消息，请发送文字" |
| `merge_forward` | 用户合并转发多条消息 | 通过 API 获取被转发的子消息列表，将每条子消息按其类型转换后用 Markdown 引用格式（`>`）拼接 |
| `share_chat` | 用户分享群卡片 | 提取群名称和 `chat_id`，格式化为：`[群名称](chat_id: oc_xxx)` |
| `share_user` | 用户分享个人名片 | 提取用户名称和 `user_id`，格式化为：`[用户名称](user_id: ou_xxx)` |
| `sticker` | 用户发送表情包 | 忽略，不处理 |
| `location` | 用户发送位置 | 提取地名和经纬度，格式化为：`位置: 地名 (经度, 纬度)` |

#### @mention 处理

`text` 和 `post` 消息中的 @mention 必须完整保留语义：
- 飞书事件的 `mentions[]` 数组包含每个被 @ 用户的 `key`（如 `@_user_1`）、`name` 和 `id`
- 转换时将占位符 `@_user_1` 替换为 `[用户名称](用户ID)`
- 这些信息必须完整传递给 Agent，因为用户可能基于 @mention 发出指令（如"帮我找 @张三 要文件"）

#### 附件统一管理

所有下载的文件（图片、文件、视频）统一保存到工作区的 `attachments/` 目录：
- 文件命名：直接使用原始文件名
- 重名冲突解决：如果文件名已存在，自动追加序号（如 `report.xlsx` → `report (1).xlsx` → `report (2).xlsx`），与 macOS 下载行为一致
- 通过 Agent 的标准附件机制传递，不散落在工作区其他位置

#### 引用消息处理

当用户的消息包含引用（`parent_id` 存在）时：

1. 系统应该通过 `GET /open-apis/im/v1/messages/:parent_id` 获取被引用消息的内容
2. 将被引用消息按其类型转换为文本，并使用 XML 标签包裹在用户消息前面：

```
<quoted_message sender="发送者名称" time="2026-03-28 10:30">
被引用的消息内容（按上述转换规则处理）
</quoted_message>

用户当前发送的消息内容
```

3. 如果被引用消息包含文件或图片，同样下载到 attachments 并作为附件传递
4. 如果被引用消息已被撤回（`deleted: true`），标注为 `[消息已撤回]`

#### 合并转发消息处理

当收到 `merge_forward` 消息时：

1. 通过 API 获取合并转发中的子消息列表
2. 按时间顺序遍历每条子消息，按其 `msg_type` 应用对应的转换规则
3. 将所有子消息拼接为 Markdown 引用格式：

```
> **发送者A** (10:30):
> 子消息1内容
>
> **发送者B** (10:31):
> 子消息2内容
```

4. 子消息中的附件（图片、文件）同样下载到 attachments 并传递

#### 验收标准

1. 当用户发送上述任一支持类型的消息时，系统应该按对应规则转换并发送给 Agent
2. 当用户发送不支持的消息类型（仅 audio）时，系统应该回复友好提示
3. 当用户的消息引用了另一条消息时，系统应该获取被引用消息内容并组装为上述格式
4. 当用户发送合并转发消息时，系统应该获取子消息列表并逐条转换
5. 当飞书事件包含已处理过的 `message_id` 时，系统应该忽略重复事件
6. 所有下载的附件应统一保存到 attachments 目录，并正确处理重名

### 需求 3 - 回复消息

**用户故事：** 作为用户，我想要在飞书中看到 Agent 的回复，以便了解任务执行进度和结果。

#### 验收标准

1. 当 Agent 完成一轮回复时，系统应该通过 `client.im.v1.message.reply()` 引用用户原始消息进行回复，以纯文本格式发送
2. 当回复内容超过飞书单条消息长度限制时，系统应该自动分段发送，后续分段使用 `client.im.v1.message.create()` 发送到同一对话
3. 当 Agent 开始处理消息时，系统应该在用户消息上添加一个"键盘输入中"的 Reaction 表情（如 `TYPING` 或 `OnIt`），表示正在处理；当回复完成后，移除该 Reaction
4. 当 Agent 执行出错时，系统应该将错误信息发送给用户

### 需求 4 - 固定会话

**用户故事：** 作为用户，我想要飞书机器人始终使用一个固定的会话，无需管理多会话，一直连续对话即可。

#### 验收标准

1. 当飞书机器人启动时，系统应该创建或恢复一个固定的专属会话（固定 chatId，如 `feishu-bot`）
2. 当用户通过飞书发送任何消息时，消息都归入这个固定会话
3. 当 Electron 应用重启后，系统应该恢复这个固定会话的 Agent 状态，继续之前的对话上下文
4. 该固定会话在桌面端会话列表中应该可见，用户可以在桌面端查看飞书对话的历史

### 需求 5 - 配置管理

**用户故事：** 作为用户，我想要在桌面端配置飞书机器人的参数，以便控制机器人的启停。

#### 验收标准

1. 当用户在设置页面中，系统应该提供飞书机器人的配置项：App ID、App Secret
2. 当用户填写配置并启用飞书机器人后，系统应该立即建立 WebSocket 连接
3. 当用户关闭飞书机器人功能时，系统应该断开 WebSocket 连接
4. 当配置项不完整时，系统应该禁用启动开关并提示缺少哪些配置
