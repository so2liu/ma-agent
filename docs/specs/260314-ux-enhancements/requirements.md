# 需求文档

## 简介

本需求覆盖 4 个用户体验增强功能：拖拽文件交互（#63）、全文搜索历史对话（#64）、长对话性能优化（#71）、错误体验优化（#72）。目标用户为非技术知识工作者，所有交互需要零认知负担。

---

## 需求

### 需求 1 - 拖拽文件到输入框（#63）

**用户故事：** 作为非技术用户，我想要直接把文件拖进应用窗口，以便快速附加文件开始对话，而不需要点击附件按钮手动选择。

#### 验收标准

1. 当用户将文件拖拽到应用窗口上方时，系统应该显示全屏半透明 drop zone overlay，提示"松开以添加文件"
2. 当用户松开文件时，系统应该将文件添加到当前输入框的附件列表中，不自动发送
3. 当用户拖拽文件离开窗口或取消拖拽时，系统应该隐藏 drop zone overlay
4. 当用户同时拖入多个文件时，系统应该将所有文件一并添加到输入框附件列表
5. 当拖入的文件超过 32MB 限制时，系统应该显示友好提示并拒绝该文件
6. 当拖入文件后，如果当前处于已有对话页面，文件应该添加到当前对话的输入框
7. 当拖入文件后，如果当前处于新对话页面，文件应该添加到新对话的输入框
8. 当拖入特定类型文件时，系统应该在输入框预填推荐 prompt：
   - `.xlsx` / `.csv` → "您想对这个表格做什么？分析数据、生成图表、还是其他？"
   - `.docx` / `.pdf` → "您想对这个文档做什么？总结、翻译、提取信息、还是其他？"
   - 图片文件 → "您想对这张图片做什么？描述内容、提取文字、还是其他？"
   - 其他/多种类型混合 → 不预填

### 需求 2 - 全文搜索历史对话内容（#64）

**用户故事：** 作为用户，我想要搜索所有历史对话的消息内容，以便快速找到"之前那个帮我处理过 XX 的对话"。

#### 验收标准

1. 当用户在侧边栏搜索框输入关键词（≥ 2 个字符）时，系统应该同时搜索对话标题和消息内容
2. 当搜索完成时，系统应该将结果分为"标题匹配"和"内容匹配"两组展示
3. 当显示内容匹配结果时，每条结果应该包含：对话标题 + 匹配消息片段（高亮关键词）+ 时间
4. 当用户点击搜索结果时，系统应该跳转到对应对话
5. 当搜索结果超过 50 条时，系统应该截断并提示"还有更多结果，请缩小搜索范围"
6. 当用户输入时，系统应该对搜索请求做 300ms 防抖处理，避免频繁搜索
7. 搜索应该在 main process 执行，遍历 conversations JSON 文件，不阻塞 UI

### 需求 3 - 长对话性能优化（#71）

**用户故事：** 作为用户，我想要在长对话（100+ 消息）中也能流畅滚动和操作，而不是遇到卡顿。

#### 验收标准

1. 当对话包含 100+ 消息时，消息列表应该使用虚拟滚动，只渲染可视区域的消息 DOM 节点
2. 当用户滚动消息列表时，帧率应该 > 55fps
3. 当新消息到达时，如果用户已在底部，列表应该自动滚动到最新消息
4. 当用户向上滚动查看历史时，新消息到达不应该强制滚动到底部
5. 当代码块超过 200 行时，系统应该默认折叠并显示"展开完整代码"按钮
6. 当对话切换时，渲染耗时应该 < 200ms

### 需求 4 - 错误体验优化（#72）

**用户故事：** 作为非技术用户，我想要在遇到 AI 服务异常时看到友好的中文提示和自动恢复，而不是看到难以理解的技术错误信息。

#### 验收标准

1. 当 API 返回 429 限流时，系统应该自动重试（指数退避：2s→4s→8s，最多 3 次），并在界面显示倒计时提示"AI 正忙，将在 X 秒后自动重试…"
2. 当 3 次重试全部失败后，系统应该显示"服务暂时繁忙，请稍后再试"并提供手动重试按钮
3. 当网络断开时（`navigator.onLine` 为 false 或 `offline` 事件触发），系统应该在界面显示离线状态提示
4. 当网络断开时，发送按钮应该被禁用，防止用户发送失败的消息
5. 当网络恢复时（`online` 事件触发），离线提示应该自动消失，发送按钮恢复可用
6. 当 API 返回 401/403 时，系统应该显示"API 密钥无效或已过期"并提供跳转到设置页的按钮
7. 当发生未知错误时，系统应该显示友好的中文描述 + "复制错误详情"按钮
8. 所有错误提示不得暴露技术性 stack trace
